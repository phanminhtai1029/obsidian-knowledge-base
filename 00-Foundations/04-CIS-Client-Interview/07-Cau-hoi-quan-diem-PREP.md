# Câu hỏi quan điểm (Opinion) — PREP

> [!summary] TL;DR
> Câu hỏi quan điểm ("What do you think…", "In your opinion…", "Which is more important…") không có đáp án "đúng/sai" — khách kiểm tra **khả năng lập luận & chính kiến**. Trả lời theo **PREP**: **P**oint (nêu thẳng quan điểm) → **R**eason (lý do) → **E**xample (ví dụ thật) → **P**oint (chốt lại). Quan trọng: **có lập trường rõ ràng** (đừng "tùy trường hợp" mơ hồ), nhưng thể hiện sự **cân nhắc** chứ không cực đoan.

---

## 1. PREP chi tiết ⭐⭐⭐

| Chữ | Trả lời gì | Lưu ý |
|-----|------------|-------|
| **P** – Point | Nêu **thẳng** quan điểm | Đừng vòng vo; chọn 1 lập trường |
| **R** – Reason | Vì sao bạn nghĩ vậy | Lý do logic, có chiều sâu |
| **E** – Example | Dẫn chứng thật minh họa | Từ kinh nghiệm bản thân càng tốt |
| **P** – Point | Khẳng định lại | Đóng khung, để lại ấn tượng |

```
★ Insight ─────────────────────────────────────
• "Có chính kiến" KHÔNG đồng nghĩa "cực đoan". Câu trả lời quan điểm tốt nhất
  thường: chọn một bên RÕ RÀNG, nhưng thừa nhận đánh đổi của bên kia ("dù giao
  nhanh có lợi ngắn hạn, tôi vẫn ưu tiên clean code vì…"). Khách thấy bạn vừa
  quyết đoán vừa cân nhắc đủ chiều.
• Example trong PREP biến quan điểm thành trải nghiệm sống: ai cũng nói được
  "clean code quan trọng", nhưng kể được một lần code ẩu khiến mình trả giá =
  quan điểm có "trọng lượng" thật.
─────────────────────────────────────────────────
```

---

## 2. Bài mẫu của bạn — "clean code vs giao nhanh" (Source-CIS.docx)

**Câu hỏi:** *"In your opinion, what's more important: writing clean code or delivering features quickly? Why?"*

> **[Point]** *"I believe writing clean code is more important in the long term."*
> **[Reason]** *"While delivering features quickly may provide short-term gains, poorly structured code increases technical debt, making future development slower and more error-prone."*
> **[Example]** *"In one of my projects, we rushed to implement features without refactoring. Later, adding a simple new function required modifying multiple parts of the system, which caused unexpected bugs. After restructuring, development became much faster and more stable."*
> **[Point]** *"Therefore, I think clean code enables sustainable speed, even if it requires slightly more effort initially."*

→ ✅ PREP chuẩn; có thừa nhận đánh đổi ("short-term gains"); Example thật; chốt bằng câu súc tích ("clean code enables sustainable speed"). Nối khái niệm **technical debt** → [[../03-Agile-Scrum/06-Technical-Debt]].

---

## 3. Các câu opinion hay gặp

| Câu hỏi |
|---------|
| Điều gì làm nên một **team player giỏi** trong dự án tech? |
| **Clean code** hay **giao tính năng nhanh** — cái nào quan trọng hơn? |
| Bạn coi trọng phẩm chất nào nhất ở một **team lead/manager**? |
| Khi có nhiều lựa chọn, bạn **quyết định dùng công nghệ nào** ra sao? |

> [!tip] Với câu "chọn công nghệ nào", PREP + tiêu chí cụ thể (yêu cầu dự án, hệ sinh thái, kinh nghiệm team, cộng đồng/hỗ trợ) sẽ ghi điểm "tư duy kỹ sư".

---

## 4. Tự kiểm tra

1. PREP là gì? *(Point – Reason – Example – Point)*
2. Câu hỏi opinion kiểm tra điều gì? *(khả năng lập luận & chính kiến, không có đúng/sai)*
3. Vì sao nên thừa nhận đánh đổi của bên kia? *(thể hiện quyết đoán nhưng cân nhắc đủ chiều)*
4. Example trong PREP có tác dụng gì? *(tạo "trọng lượng" — quan điểm từ trải nghiệm thật)*
5. Vì sao kết bằng Point thứ hai? *(đóng khung, để lại ấn tượng cuối)*
